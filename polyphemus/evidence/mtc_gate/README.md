# MTC Gate Receipts

Generated by `tools/mtc_pre_deploy_gate.py` when invoked with `--write-receipt`.
Consumed by `predeploy.sh` step 2 (MTC gate receipt check) to block LIVE deploys
without fresh statistical evidence.

## Receipt filename

`{segment_key}_{generated_at}.json` where:
- `segment_key = trades_{strategy}` for per-strategy gates (signal_bot, pair_arb, weather_arb)
- `segment_key = cycles_{asset}_{window_duration_secs}` for accumulator gates

## When predeploy.sh requires a receipt

| VPS `.env` flag     | Required segment     |
| ------------------- | -------------------- |
| `DRY_RUN=false`     | `trades_signal_bot`  |
| `ACCUM_DRY_RUN=false` | `cycles_btc_300`   |
| `WEATHER_DRY_RUN=false` | _warn only, no gate yet_ |

Any live-bound deploy without a PASS receipt younger than 7 days is blocked.

## Generating a receipt

```bash
python3 -m polyphemus.tools.mtc_pre_deploy_gate \
  --source trades --db polyphemus/data/performance.db \
  --strategy signal_bot --lookback-days 30 \
  --write-receipt polyphemus/evidence/mtc_gate
```

Exit code 0 = PASS (receipt useful for deploy gate). Exit code 1 = FAIL (receipt
still written so the failure is auditable, but predeploy.sh will reject it).

## Why receipts live in git

Each receipt is a few KB of aggregate statistics (no PII, no wallet addresses).
Committing them produces a permanent audit trail: every live-bound deploy is
pinned to a specific statistical verdict, and `git log polyphemus/evidence/mtc_gate/`
reconstructs the go/no-go history.
